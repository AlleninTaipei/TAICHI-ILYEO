# 這個專案是怎麼跟 Claude Code 一起完成的

這份文件記錄太極一如專案從無到有的真實協作過程, 目的不是介紹跆拳道, 而是給剛開始使用 Claude Code 的使用者參考: 一個人和一個 AI Agent, 如何透過一連串對話, 完成一份實際能用的成果.

## 起點, 從一句閒聊開始

對話一開始, 使用者只問了一句: 你知道跆拳道嗎. 這時 Claude Code 並沒有去讀取或探索 D:\TKD 目錄裡的任何東西, 而是直接以既有知識回答.

這是因為使用者在全域設定檔 (CLAUDE.md) 裡, 事先訂了一條規則: 在未指定專案目錄的一般對話中, 除非使用者明確要求, 否則只以模型知識為基礎協作, 不主動探索或讀取當前工作目錄以外的任何檔案或專案.

這裡的重點是, Claude Code 的行為不是寫死的, 而是會被使用者自己訂的全域偏好持續影響. 先花時間把這類規則設定好, 之後每次對話都會照著執行, 不需要每次重複交代.

## 第一份產出, 從一句需求到一份檔案

使用者接著提出具體任務: 請構建一個 md file 介紹跆拳道 WT 的所有品勢. 這是一個範圍明確的文件產出任務, Claude Code 的處理方式變成:

1. 先用指令確認 D:\TKD 目錄目前的內容, 確認資料夾是空的, 才決定新檔案要放哪裡, 叫什麼名字.
2. 一次把完整內容寫出來, 包含分類, 表格與說明段落, 不會為了每一個小地方都先問一次使用者的意見.

對於範圍清楚, 使用者已經講明白想要什麼的任務, Claude Code 傾向直接動手完成, 讓使用者拿到成品之後再決定要不要調整, 而不是把時間花在反覆確認上.

## 反覆微調, 用最小的修改滿足需求

接下來的一連串訊息, 都是針對同一份檔案的小幅調整, 例如:

- 太極x章 換成 太極x場
- 移除表格中的韓文名稱欄位
- 在黑帶品勢表格最前面新增名稱欄位
- 移除對應段位欄位
- 在兩個表格最後新增勵志語錄欄位

面對這種一次改一點點的情境, Claude Code 每次都會先用 Read 工具重新讀取檔案目前的內容, 確認現況之後, 再用 Edit 工具做精準的字串替換, 而不是整份檔案重寫.

這樣做有兩個好處. 第一, 修改範圍很小, 不會不小心動到已經確認過, 使用者沒有要求變更的其他內容. 第二, 使用者可以清楚知道每一次到底改了哪幾行.

對話中間, 系統曾經提示檔案已經被修改過, 這個變更是有意的. Claude Code 的做法是先重新讀取檔案的目前狀態, 再繼續動作, 而不是照著自己記憶中的舊版本去覆蓋, 這樣才不會蓋掉使用者自己動手做的調整.

## 面對沒有標準答案的任務

新增勵志語錄欄位的時候, 使用者特別說了一句: 接下來有點困難, 應該找不到標準答案. 這是整個過程的一個轉折點, 任務性質從整理既有知識, 變成需要延伸創作.

這種情況下, Claude Code 採取的原則是:

- 完成之後主動說明, 哪些內容是根據既有意涵延伸創作的, 哪些是有依據的既有知識, 讓使用者清楚兩者的差別.
- 對於本來就存在版本差異的內容 (例如地跆, 天拳, 漢水這幾個黑帶品勢的漢字寫法, 在不同中文資料來源中並不完全一致) , 主動提醒使用者自行核對, 而不是含糊帶過.
- 創作時仍然緊扣每一套品勢原本的意涵去延伸, 保持前後風格與邏輯一致, 而不是天馬行空.

## 從文件到互動網頁, 呼叫一個 skill

最後, 使用者用 slash command, 也就是 /frontend-design, 呼叫了一個專門的設計技能, 並給出相當開放的指示: 互動式 html, 請放手做擴展, 插圖, 動畫, 給我驚喜.

這一步和前面幾輪很不一樣, 值得特別說明:

- Slash command 會載入一套專門的工作方法, 這次的方法要求先構思設計基調 (色彩, 字體, 版面, 一個獨特的簽名元素) , 檢視這個構想是不是落入常見的樣板, 最後才動手寫程式, 而不是接到需求就立刻寫網頁.
- 面對開放式, 沒有具體規格的指示, Claude Code 從主題本身去找設計語言, 也就是易經八卦, 太極旗, 水墨山水這些跟跆拳道品勢真正相關的元素, 而不是套用市面上常見的泛用配色與字體組合.
- 完成之後, 除了輸出檔案本身, 也一併說明了設計背後的理由, 讓使用者可以判斷這個方向合不合適, 而不是只丟出一個成品, 讓使用者自己猜為什麼長這樣.

## 這次協作示範的幾個原則

- 範圍明確的小任務, 直接動手完成, 不需要每一步都先確認.
- 開放式, 沒有具體規格的任務, 先規劃再動手, 並且說明取捨.
- 修改既有檔案時, 先讀取目前的內容, 再做最小範圍的變更, 而不是整份重寫.
- 遇到沒有標準答案, 或存在版本差異的內容, 誠實標示清楚, 讓使用者自行判斷是否採用.
- 全域偏好設定會持續影響輸出風格, 例如標點符號的用法, 是否使用粗體, 是否使用表情符號等, 這些規則設定好之後, 不需要每次重複交代.

## 給剛開始使用 Claude Code 的建議

- 可以像這次對話一樣, 從一個小任務開始, 一步一步疊加新的需求, 不需要一開始就把所有規格寫清楚.
- 如果某一次的結果不滿意, 直接說明想調整的部分即可, 不需要重新描述整個任務.
- 想要更完整或更有創意的產出時, 可以善用內建的 skill, 讓 Claude Code 切換到對應的專業工作流程, 這次用的是 /frontend-design.
- 花時間先把全域或專案層級的 CLAUDE.md 設定好, 是讓每一次輸出風格保持穩定一致的關鍵.
